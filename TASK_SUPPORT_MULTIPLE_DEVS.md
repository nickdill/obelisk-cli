# PROMPT

The next phase of this will likely be supporting multiple devs working on teams to manage obelisks. What's a good way to approach that? For example I deploy to an obelisk, then my teammate to somehow do `obelisk list` and be able to see that same obelisk and somehow have permission and authentication to access the obelisk - but the catch is without logging in since this is open source, instead like have a secret key and have the obelisk track allowed secret keys something. Plan and outline a suggested system that is simple but incredibly secure and good for me to use.

# ANSWER

To support collaborative team environments on a self-hosted, open-source project without forcing users onto a centralized SaaS login, you need a decentralized, cryptographic trust model.Instead of a traditional database-backed "username and password" system, you can model this after SSH Authorized Keys. The Obelisk itself acts as the source of truth for who is allowed to talk to it, using asymmetric cryptography (public/private key pairs) or a shared-secret rotation.Here is a simple, incredibly secure blueprint for a team-based, server-centric authentication system for your framework.The Concept: "The Cryptographic Doorbell"Every developer generates a local identity. The Obelisk holds a list of public identities that are allowed to modify it. When a teammate runs mega deploy or mega list, they sign their request with their private key, and the Obelisk verifies it.[ Developer A (Admin) ] ───(Invites via Public Key)───► [ Obelisk Server ]
                                                              ▲
[ Developer B (Teammate) ] ───(Signs request with PrivKey)────┘
Here is how to structure the system in three steps: Identity, Onboarding, and Request Verification.1. Identity Generation (The Local Setup)When a developer installs the CLI, they need a unique cryptographic identity. You can automate this behind a simple setup command.The Command: mega init-user (or run automatically on first use).Under the Hood: The CLI generates an ED25519 key pair (the same modern algorithm SSH uses).Private Key: Stored safely in the user's home directory (e.g., ~/.config/megalisk/id_ed25519). Never leaves their machine.Public Key: Safe to share with anyone. This is their "Username".2. Team Onboarding (The Trust Handshake)How does Developer B get access to Developer A's Obelisk? You use a simple Invite Code or Direct Append workflow.Approach: The allow-key CommandIf Developer A already has access to the Obelisk, they can authorize Developer B directly from the CLI.Developer B copies their public key and slacks it to Developer A:Bashmega identity --public
# Output: mega-pub-11a2b3c4d5e6...
Developer A runs an administrative command to push that key to the Obelisk:Bashmega admin allow-key mega-pub-11a2b3c4d5e6... --name "Teammate Bob"
The Obelisk appends this key to its internal allowed_keys.json file.3. How a Secure Request Works (mega list)Because there is no "session login," every single CLI command sent to the remote Obelisk must prove it has permission in real-time.To prevent hackers from intercepting requests (Replay Attacks), you use a Cryptographic Challenge-Response or a Signed Timestamp.The Flow for mega list:The Request: The CLI builds a payload containing the current timestamp and the intent:JSON{
  "command": "list",
  "timestamp": 1783684000
}
The Signature: The CLI uses the developer's Private Key to sign that JSON payload. It attaches the signature and the developer's Public Key to the HTTP headers.The Obelisk Verification: * The Obelisk receives the request.Step A (Time Check): It checks if the timestamp is within the last 30 seconds. If it's old, it rejects it (prevents replay attacks).Step B (Identity Check): It looks at the incoming Public Key and checks if it exists in its local allowed_keys.json.Step C (Cryptographic Check): It uses the Public Key to verify that the signature matches the payload. If it matches, the Obelisk trusts it completely and executes the command.4. Discovering the Obelisks (mega list)If there is no central server, how does Developer B's CLI even know where the Obelisk is located (its IP address or domain) to run mega list?You have two excellent choices for an open-source tool:Option A: The Project-Bound Config (Simplest)Keep the Obelisk’s remote endpoint URL inside the project repository's obelisk.yml file.When Developer B clones the Git repo, the CLI automatically reads endpoint: https://obelisk.myteam.com.They type mega list, the CLI fires the signed request to that URL, and the Obelisk returns the active modules.Option B: Local ProfilesIf you don't want server URLs committed to Git, Developer B can save the target once locally:Bashmega target add company-obelisk https://obelisk.myteam.com
mega list --target company-obelisk
Summary of the UX ArchitectureCommandWhat happens under the hoodmega init-userGenerates asymmetric crypto keys locally on the dev machine.mega admin allow-key [key]Authorizes a teammate's public key on a specific Obelisk.mega listReads the target URL from config, cryptographically signs a timestamped packet, and requests the module state from the Obelisk.This approach keeps everything open-source, requires zero databases on your end, eliminates password leaks, and gives developers complete, sovereign control over their own infrastructure.
